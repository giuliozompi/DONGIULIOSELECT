🛠️ Техническое задание (ТЗ) для Telegram Mini App (TMA)
Название проекта: Don Giulio Select Catalog Mini App
Цель: Создание высокотехнологичного и нативного интернет-магазина (каталога) в среде Telegram Mini App с акцентом на современный UX и интеграцию с API мессенджера.
Стек технологий: React / TypeScript / Tailwind CSS, @twa-dev/sdk для интеграции с Telegram WebApp.
1. Общие Требования к Нативности и Дизайну
1.	Нативность TMA: Обязательное использование API Telegram WebApp.
o	Theme Adaption: Полная адаптация всех элементов к светлой (themeParams.bg_color) и темной (themeParams.secondary_bg_color) темам Telegram.
o	MainButton (CTA): Главное действие на странице (добавление в корзину) должно использовать нативный WebApp.MainButton.
o	BackButton: Для навигации использовать нативный WebApp.BackButton.
o	Haptic Feedback: Использовать легкую вибрацию (impactOccurred('light'/'success')) для подтверждения ключевых действий (добавление в корзину, нажатие на Главную кнопку).
2.	Дизайн (Modern Telegram Style):
o	Минимализм: Чистый, адаптивный дизайн с большим количеством свободного пространства ("воздуха").
o	Типографика: Использование системных/нативных шрифтов и жирного шрифта для акцентов (Название, Цена).
o	Адаптивность: Полная поддержка всех мобильных разрешений (Mobile First).
2. Структура Данных (Модель Продукта)
Необходимо создать структуру данных для обычного и сборного продукта.
2.1. Модель Обычного Продукта (Product)
interface Product {
    id: string; // Уникальный артикул (внутренний)
    category: string; // Категория (Сыры / Из коровьего молока / Рассольные)
    nomenclature: string; // Номенклатура / Название (Сыр Моцарелла)
    article_internal: string; // Артикул внутренний
    article_vendor: string; // Артикул производителя
    unit_of_measure: 'кг' | 'шт' | 'мл' | string; // Единица измерения
    price_per_unit: number; // Цена за единицу (руб.)
    price_old?: number; // Старая цена (если есть скидка)
    weight_min_grams: number; // Минимальный вес для продажи (для весового товара)
    photos: string[]; // Массив URL фото (Макс. 10 шт.)
    taste_variations?: string[]; // Массив доступных Вкусов (например, ['Классический', 'С травами'])
    taste_rating_stats: { super: number, very_tasty: number, tasty: number }; // Статистика по вкусовому рейтингу
    user_selected_taste_rating?: 'супер' | 'очень вкусное' | 'вкусное' | null; // Выбор текущего пользователя
    description_short: string; // Краткое описание продукта (видимо сразу)
    description_full: string; // Полное описание продукта
    nutrition: {
        bju: { proteins: number, fats: number, carbs: number, calories: number }; // БЖУ и калорийность (на 100г/мл)
        values: string[]; // Питательные ценности (Витамин D, Кальций)
        composition: string; // Состав (полный список ингредиентов)
    };
    consumption_tips: string; // Как и с чем употреблять (сочетания)
    related_products: string[]; // Связанные продукты (по ID)
    related_recipes: string[]; // Связанные рецептуры (по ID)
    related_videos: string[]; // Ссылки на связанные видео (YouTube/Vimeo)
}

2.2. Модель Сборного Продукта (SetProduct)
•	Карточка продукта сборная (подарочный/продуктовый набор) должна содержать поле is_set_product: true.
•	Вместо вариаций "Вкус" должно быть поле set_composition: Array<{product_id: string, quantity: number}>, показывающее, из чего состоит набор.
•	Визуализация: В карточке сборного продукта состав должен быть представлен в виде аккуратного списка со ссылками на входящие в него товары.
2.3. Модель Приза (Prize)
interface Prize {
    prize_id: string; // Уникальный ID приза
    name: string; // Название приза (Например, "Сырный сэмпл", "-10% на первый заказ")
    type: 'discount' | 'product_gift' | 'delivery_coupon' | 'stars_bonus'; // Тип приза
    value: string; // Значение приза (Например, "10", "Сыр Моцарелла ID: X")
    image_url: string; // Изображение для сектора Колеса Фортуны
}

2.4. Модель Статуса Геймификации Клиента (UserGamificationStatus)
interface UserGamificationStatus {
    user_id: string; 
    spin_tokens: number; // Доступное количество вращений
    last_spin_date: Date | null; // Дата последнего вращения
    earned_prizes: Array<{ prize_id: string, claimed: boolean, expiry_date: Date }>; // Список заработанных призов
}

3. Функционал: Карточка Продукта (A. Каталог Продукции)
Карточка должна быть реализована согласно детальному современному дизайну, с использованием секций-аккордеонов для скрытия большого объема текста.
3.1. Визуальное Оформление Карточки
Секция	Элементы	Оформление / Логика
Галерея (A)	photos	Карусель на всю ширину. Горизонтальный свайп. Индикаторы (точки) внизу.
Заголовок (B)	nomenclature, category (мелким), article_internal	Название – крупно, жирно. Категория – тонким, серым шрифтом. Артикул – внизу страницы в разделе деталей или скрыт.
Цена (C)	price_per_unit, price_old, unit_of_measure	Цена – самый крупный элемент (кроме фото), цвет AccentText. Единица измерения (кг/шт) рядом. Если есть price_old, показать зачеркнутым.
Выбор (D)	taste_variations	Используются кнопки-чипы для выбора вкуса. Недоступные вкусы деактивированы.
3.2. Секции Детализации (Аккордеоны)
Для полей, содержащих большой объем текста, использовать блоки-аккордеоны (развернуть/свернуть), чтобы не перегружать экран.
1.	Краткое Описание: description_short (всегда видно).
2.	Полное Описание: description_full (скрыто под "Показать полностью...").
3.	Пищевая Ценность: Аккордеон. Содержит:
o	nutrition.bju (БЖУ и Калорийность в виде таблицы).
o	nutrition.values (Питательные ценности - маркированный список).
4.	Состав: Аккордеон. Содержит nutrition.composition.
5.	Применение: Аккордеон. Содержит consumption_tips.
3.3. Блоки Рекомендаций (Внизу Карточки)
•	Связанные Продукты: Горизонтальный скролл-лист мини-карточек товаров (related_products).
•	Связанные Рецептуры: Горизонтальный скролл-лист карточек рецептов (related_recipes).
•	Связанные Видео: Блок с превью видео (related_videos).
3.4. Главное Действие (CTA)
•	Main Button: Всегда отображается внизу экрана.
•	Текст: "Добавить
Вес/Кол−во

•	Логика: При клике вызывается HapticFeedback, текст меняется на "✅ Добавлено", и выполняется асинхронный запрос на добавление товара в корзину.
3.5. Функционал: Рейтинг Вкуса (Taste Rating)
Необходимо добавить отдельный блок для оценки вкуса продукта, используя систему чипов.
•	Механика: Использовать три кнопки-чипа (Toggle Chips) с четкой логикой выбора:
1.	"Вкусное" (самый низкий рейтинг)
2.	"Очень вкусное" (средний рейтинг)
3.	"Вообще супер" (самый высокий рейтинг)
•	Визуализация: Блок расположен сразу под названием или кратким описанием. Выбранный пользователем чип должен быть контрастно выделен, используя themeParams.button_color. Остальные чипы – в нейтральном стиле.
•	Логика: При нажатии на чип выполняется асинхронный POST-запрос на /api/product/rate-taste с ID продукта и выбранным рейтингом. Должен быть Haptic Feedback ('light').
•	Статистика: Под кнопками-чипами должна отображаться общая статистика (taste_rating_stats), например: "65% покупателей оценили как 'Вообще супер'".
4. Функционал: Каталог и Навигация (Разделение по Категориям)
Навигация должна быть реализована с помощью трехуровневой структуры категорий, как описано в ТЗ:
1.	Уровень 1: Основные разделы (Сыры, Мясные Деликатесы, Бакалея, Десерты, Посуда).
2.	Уровень 2 (Сыры): Тип молока (Коровье, Козье, Овечье, Буйволиное).
3.	Уровень 3 (Сыры): Тип сыра (Рассольные, Мягкие, Полутвердые, Твердые).
•	Реализация: Каталог должен использовать вложенные списки или табы. На каждом уровне должен отображаться список товаров, соответствующих выбранным фильтрам.
•	Отображение: Использовать карточки-превью с фото, названием и ценой, ведущие на детальную карточку продукта.
5. Функционал: Ассистент (AI Assistant)
Необходимо интегрировать функционал AI Assistant.
•	API: Для генерации текста использовать openrouter API.
•	Интерфейс: Отдельный раздел (или модальное окно) "Ассистент", где пользователь может задать вопрос.
•	Роль Ассистента: Роль — эксперт по продуктам (сочетаниям, гастрономии).
•	Контекст: Ассистент должен использовать следующие данные (моковые или контекстные) для формирования рекомендаций:
o	История пользователя (предыдущие заказы).
o	Текущая корзина (товары в корзине).
o	Связанные продукты (логика "Обычно покупают").
o	Специальные даты (моковое "День рождения" в профиле пользователя).
o	Примеры запросов: "Какое вино лучше подойдет к этому сыру Камамбер?", "Посоветуй десерт по настроению", "Что заказать к моему дню рождения?".
6. Функционал: Спецзаготовка Заказа
Необходимо добавить возможность выбора оформления заказа для подарочных/сервировочных целей.
•	Механика: В Корзине или на этапе оформления заказа должна быть опция "Спецзаготовка".
•	Опции:
o	В контейнер
o	В тарелку
o	В коробку
o	На доску
o	Без обработки (по умолчанию)
•	Каталог Сервировки: При выборе опции (кроме "Без обработки") пользователь должен перейти в отдельный, отфильтрованный каталог (Сервировочная посуда) для выбора конкретной Тарелки, Доски или Коробки.
•	Интеграция с Корзиной: Выбранный элемент сервировки добавляется в корзину как отдельный товар с нулевой стоимостью или как дополнительная услуга.
7. Функционал: Корзина, Оплата и Доставка
7.1. Корзина
•	Отображение: Список товаров, возможность изменения количества/веса (используя минимальный шаг 100гр  weight_min_grams или 1 для штучного товара).
•	Удаление: Функция удаления товара и очистки корзины.
•	Итоги: Подсчет общей суммы заказа и отображение стоимости доставки. Важно: Если заказ содержит весовые товары (unit_of_measure: 'кг'), итоговая стоимость должна быть четко обозначена как "Примерная сумма заказа".
•	Корзина должна иметь функционала дата доставки и комментари к заказу
7.2. Оплата (После подтверждения оператором)
Механика: Оплата заказа (независимо от его состава) производится только после его подтверждения и финализации стоимости оператором Back Office.
1.	Заказ Размещен: После оформления заказа клиенту показывается статус "Заказ принят, ожидается подтверждение оператором и расчет точной стоимости".
2.	Оператор Финализирует: Оператор Back Office (см. 8.2) подтверждает наличие товаров, уточняет вес (для весовых позиций) и фиксирует окончательную сумму.
3.	Оплата Клиентом: Система отправляет клиенту уведомление (моковое) с точной суммой и генерирует модальное окно для оплаты.
•	QR-код: Реализация модального окна с симуляцией оплаты по QR-коду.
•	Ссылка на оплату: Генерация ссылки (моковая) для внешней оплаты.
7.3. Доставка
•	Интеграция: Создание моковых API-слоев для интеграции с:
o	Яндекс Go: (Моковые тарифы/сроки для Москвы/МО).
o	СДЭК: (Моковые тарифы/сроки для России с учетом ограничений на продукты).
•	Доставка в подарок:
o	На этапе оформления заказа должны быть поля: От кого и Кому.
o	Эта информация должна быть сохранена вместе с заказом.
8. Функционал: Back Office / Mini-CRM (для администратора)
Этот раздел описывает функционал управления, доступный только авторизованным администраторам.
•	Доступ: Реализовать отдельный маршрут или условный рендеринг для администраторов (например, проверка userId против списка администраторов).
•	Дизайн: Простой, функциональный интерфейс, оптимизированный для быстрой обработки данных.
8.1. Управление Каталогом (CRUD)
•	Карточка Продукта: Предоставить администратору интерфейс для создания, просмотра, изменения и удаления (CRUD) всех полей Модели Обычного Продукта (Product) и Сборного Продукта (SetProduct).
•	Фото/Медиа: Функция загрузки и управления фото (photos).
•	Рейтинг Вкуса: Возможность просмотра и сброса статистики (taste_rating_stats).
8.2. Обработка Заказов
•	Список Заказов: * Табличное представление всех заказов.
o	Фильтрация/Сортировка: По статусу (Новый, В обработке, Готов к доставке, Завершен, Отменен), по дате, по клиенту.
•	Карточка Заказа: * Детальный просмотр состава, первоначальной примерной суммы, данных клиента и доставки.
•	Финализация Заказа (Критичный шаг): Оператор должен иметь функционал для:
o	Уточнения веса: Внесение точных значений веса для весовых товаров.
o	Расчета Итоговой Суммы: Автоматический или ручной расчет окончательной стоимости.
o	Подтверждения: Кнопка "Финализировать и Отправить ссылку на оплату". Это действие переводит статус заказа в "Ожидает оплаты" и запускает процесс уведомления клиента (см. 7.2).
•	Управление Статусом: Набор кнопок для изменения статуса заказа (после оплаты).
8.3. Отслеживаемость Закупок Клиентов
•	Просмотр Клиентов: Список всех пользователей, совершавших покупки.
•	Профиль Клиента (Mini-CRM):
o	Отображение userId (для идентификации в Telegram).
o	LTV (Lifetime Value): Общая сумма, потраченная клиентом.
o	История Заказов: Список всех предыдущих заказов с датами и итоговыми суммами.
8.4. Напоминания и Mini-CRM Логика
•	Интерфейс Напоминаний: Отдельная вкладка для работы с "спящими" или целевыми клиентами.
•	Оповещение Оператора: При обнаружении "спящего" клиента система визуально оповещает оператора (например, через уведомление или счетчик на вкладке).
•	Логика Выборки (Моковая):
1.	"Спящие" клиенты: Клиенты, не делавшие заказ ≥30 дней.
2.	"Повторные" клиенты: Клиенты, которые покупали скоропортящиеся товары (Сыры, Деликатесы) ≥14 дней назад.
•	Действие: Предоставить форму для написания текста напоминания и кнопку "Напомнить". Нажатие на кнопку должно инициировать отправку напоминания выбранному клиенту в один клик (моковая логика).
9. Функционал: Геймификация (Колесо Фортуны и Призы)
Этот раздел описывает механику вовлечения клиентов через игру и призы.
9.1. Механика "Колесо Фортуны" (Wheel of Fortune)
•	Интерфейс: Отдельный, полноэкранный раздел с интерактивным, красочным Колесом Фортуны.
•	Сектора Колеса: Напрямую связаны с Актуальным Каталогом Призов (см. 9.2). Колесо должно визуально отражать 6-8 актуальных призов, плюс один сектор "Не повезло" / "Следующий раз".
•	Кнопка "Вращать": Расположена в центре или под колесом. Неактивна (disabled), если у пользователя
spin_tokens=0
$$$$$$$$.
•	Логика Вращения:
1.	Нажатие на "Вращать" -> Haptic Feedback ('heavy').
2.	Отправка запроса: POST /api/gamification/spin.
3.	Backend (моковый) определяет приз (на основе весов призов) и уменьшает spin_tokens.
4.	Анимация вращения с плавным замедлением и остановкой на выигранном призе.
5.	Показ Модального окна с поздравлением, названием приза и кнопкой "Перейти в мои призы".
9.2. Управление Доступом (Токены Вращения)
•	Условия Начисления Токенов (Моковые):
o	Первая покупка: +1 токен.
o	Покупка на сумму: За каждые ≥5000₽ в одном чеке — +1 токен.
o	Ежемесячный вход: +1 токен (с лимитом 1 раз в месяц).
•	Отображение: Количество доступных вращений (spin_tokens) должно быть видно в верхней части экрана геймификации.
9.3. Каталог Призов
•	Управление Каталогом (Back Office): Администратор должен иметь возможность обновлять список активных призов (Prize Model) через Back Office.
•	Клиентский Каталог ("Мои Призы"): Отдельный раздел, где пользователь видит:
o	Список всех выигранных призов (earned_prizes).
o	Статус: "Не использован", "Использован" или "Срок истек".
o	Кнопка "Использовать": При нажатии применяется соответствующая скидка (например, промокод копируется в буфер) или добавляется бесплатный товар в корзину.
o	Срок действия: Каждый приз должен иметь моковую дату истечения срока действия (expiry_date).
10. Функционал: Контактные данные и Профили Доставки
Цель: Сбор и хранение необходимой информации для доставки и связи с клиентом, с поддержкой возможности оформления заказа на разные адреса и разных получателей (даже если клиент оформляет заказ с одного аккаунта).
10.1. Требуемые Контактные Данные
Клиент должен предоставить следующие стандартные данные на этапе оформления заказа:
1.	ФИО Получателя (Required): Полное имя или имя для обращения.
2.	Контактный телефон Получателя (Required): Телефон для связи курьера/оператора с получателем.
3.	Полный адрес доставки (Required): Город, улица, номер дома, корпус/строение.
4.	Детали адреса (Optional): Номер квартиры/офиса, подъезд, этаж, код домофона.
5.	Email (Optional): Для дополнительных уведомлений.
10.2. Механика Управления Профилями Доставки
•	Модель Профиля Доставки (DeliveryProfile): Каждый уникальный набор (ФИО Получателя + Телефон + Адрес) сохраняется как отдельный профиль.
•	Сохранение и Выбор:
o	Клиент должен иметь возможность сохранять несколько Профилей Доставки.
o	На этапе оформления заказа должен быть предусмотрен удобный интерфейс (например, выпадающий список или список карточек) для выбора одного из сохраненных профилей.
o	При вводе нового адреса система должна предложить сохранить его как новый профиль для будущих заказов.
•	Использование: Выбранный профиль доставки привязывается к текущему заказу.
