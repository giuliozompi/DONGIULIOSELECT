Интеграция через API: Яндекс Go Доставка vs Яндекс Доставка
Ниже — практическое сравнение двух сервисов и как к каждому подключаться с точки зрения разработчика интеграции.

Коротко о различиях
Яндекс Доставка (логистическая платформа, известна как Delivery/Yandex Delivery)
B2B‑сервис последней мили и межгород: курьерка, доставка день‑в‑день/на следующий, забор со склада/точки, слоты, массовые отгрузки.
Богатое API для интернет‑магазинов, маркетплейсов, 3PL, омниканала.
Функционал: расчёт тарифов, создание заказов, трекинг, расписание курьеров, возвраты, наложенный платеж, акты, вебхуки, роутинг.
Тип интеграции: сервер‑сервер, продакшн‑уровень, документы и статусы.
Яндекс Go Доставка (он‑деманд, интеграция с курьерами Яндекс Go)
Фокус на “сейчас”/в течение часа/день‑в‑день из магазина/кафе/пункта: быстрая курьерская доставка, часто для ритейла/HoReCa/D2C.
Простое API для вызова курьера по требованию, трекинг по ссылке.
Модель ближе к такси‑стайлу: быстрый расчёт, привлечение ближайшего исполнителя.
Тип интеграции: легче старт, меньше документооборота, упор на скорость.
Если у вас интернет‑магазин с партиями заказов, SLA, слотами и возвратами — чаще подходит Яндекс Доставка. Если нужны точечные срочные доставки/доставка по клику — Яндекс Go Доставка.

Основные возможности API
Яндекс Доставка (Delivery)
Расчёт стоимости и сроков: передаёте параметры груза, адреса, сервисы (температурный режим, НП, частичный выкуп).
Создание и управление заказами: создание, изменение окна, отмена, печать этикеток.
Трекинг: статусы, ETA, движение курьера, вебхуки о смене статусов.
Курьеры и расписание: забор отгрузок, слоты, графики.
Документооборот: акты сдачи‑приёмки, отчётность, возвраты.
Массовые операции: создание батчей, мульти‑посылки.
География: внутри города и межгород (в зависимости от тарифов и партнёров).
Типичные эндпоинты:

POST /delivery/calculate — калькуляция
POST /delivery/create — создать заказ/отгрузку
GET /delivery/{id}/status — статус
Webhooks: /webhooks/status-changed
(Точные пути зависят от версии API; провайдер может использовать v1/v2 и разные маршруты для “orders”, “shipments”, “batches”.)
Аутентификация:

OAuth2/токен сервиса или API‑ключ (в зависимости от подключения). Часто используется Bearer‑токен в заголовке Authorization.
Идёмпотентность: заголовок Idempotency-Key для безопасных повторов.
Яндекс Go Доставка
Быстрый расчёт цены и ETA для он‑деманд заказа.
Создание заказа с координатами/адресами “откуда–куда”, особыми инструкциями, контактами.
Трекинг курьера в реальном времени, ссылки для клиента.
Простые вебхуки о смене статусов (принят, в пути, у точки, доставлен/отменён).
Опционально: оплата по безналу для бизнеса, промокоды/ограничения через бизнес‑кабинет.
Типичные эндпоинты:

POST /claims/calculate — расчет
POST /claims/create — создать заявку
GET /claims/{id} — получить состояние
POST /claims/cancel — отменить
Webhooks: /webhooks/claim-status
(Набор и имена зависят от текущей версии “b2b-taxi/drive/delivery” API для Go.)
Аутентификация:

OAuth‑токен/сервисный токен B2B‑кабинета Яндекс Go.
Часто также используется идемпотентность по заголовку.

Ключевые различия для интегратора
Модель выполнения:
Go: быстрее найти ближайшего исполнителя, on-demand; динамичная цена.
Delivery: больше вариантов тарифов и сценариев, стабильные SLA, планирование.
Документы и финансы:
Go: минимальный документооборот, простой биллинг через бизнес‑аккаунт.
Delivery: акты, закрывающие документы, NП, возвраты, отчёты.
Масштабирование:
Go: точечные вызовы, меньше батчевости.
Delivery: партии, multi‑stop, multi‑parcel, интеграция со складом/WMS.
География и сервисы:
Go: в основном городская курьерка здесь‑и‑сейчас.
Delivery: город/межгород, доп. услуги, температурные режимы (если подключены партнёры).
Техподдержка и SLA:
Delivery чаще предусматривает расширенную поддержку B2B и SLA на статусы/сроки.
Go делает акцент на быстром отклике и простоте.
Типичные ошибки и советы
Неиспользование идемпотентности → дубли заказов при сетевых ретраях.
Отсутствие валидации габаритов/веса → отказ при назначении курьера.
Неверная обработка вебхуков → расхождение статусов в вашей системе.
Неучтённые окна времени и входные инструкции → задержки или отмены.
Недостаточное логирование запросов/ответов → сложно дебажить инциденты.
Рекомендации:

Введите корреляционный request_id на весь поток: калькуляция → создание → статусы.
Обязательно переотправляйте вебхуки (dead-letter queue) и реализуйте идемпотентные обработчики.
Делайте fallback: если Go не находит курьера в нужное время, автоматически пробуйте Delivery‑тариф, и наоборот, если ваш бизнес‑процесс это допускает.
Следите за лимитами RPS и экспоненциальными ретраями с jitter.
Что необходимо от вас для старта интеграции
Выбор сервиса: Go, Delivery или гибрид.
Сценарии: срочная доставка сейчас/слоты/межгород, НП, возвраты, температурный режим.
Данные: формат адресов, координаты, вес/габариты, цена, контактные данные.
Процесс: кто создаёт заказ (витрина/OMS/WMS), когда рассчитываем цену, когда печатаем ярлык.

Короткий ответ: потому что “Яндекс Доставка” для малого/среднего бизнеса и on‑demand курьерки сейчас авторизуется через общий бизнес‑кабинет Яндекс Go. Бренды и личные кабинеты частично объединены, поэтому вход по ряду “доставочных” ссылок редиректит в Яндекс Go.

Чуть подробнее, что это значит на практике:

Единая авторизация. У B2B‑направлений курьерской доставки Яндекса используется общий аккаунт и SSO через Яндекс Go for Business. Поэтому:
Ссылки “войти в Яндекс Доставку” часто ведут на домены Go и/или редиректят на общий SSO.
Внутри кабинета вы увидите разделы/продукты доставки, тарифы, заявки (claims), вебхуки и настройки API.
Разделение по продуктам осталось на уровне API и функционала:
Go Доставка: он‑деманд курьеры “здесь и сейчас”, создание заявок/claims.
Яндекс Доставка: плановые отгрузки, слоты, партии, документы. Доступ к нему тоже может открываться через общий вход, но дальше вы попадаете в соответствующий продукт/раздел.
Что проверить, если вас “упорно” ведёт в Go:

Роль и доступы в аккаунте. Возможно, вашему логину не выдан доступ к разделу “Яндекс Доставка” — нужен админ вашей организации, чтобы включить нужный продукт/права.
Регион и договор. Некоторые разделы видны только после заключения договора или подключения тарифа в вашем регионе.
Куки и SSO. Попробуйте:
открыть ссылку в приватном окне,
выйти из всех аккаунтов Яндекса и войти снова под нужной организацией,
выбрать нужный бизнес‑аккаунт в переключателе организации в шапке кабинета.
Прямой заход в разделы. Внутри Go‑кабинета найдите пункты меню, связанные с “Доставкой/Delivery”, “Заявки/Claims”, “Интеграции/API”, “Вебхуки”.
Если у вас конкретная цель (например, включить API Яндекс Доставки, а не Go), подскажите:

какая организация/ИНН,
какой сценарий (плановые отгрузки со склада или срочные курьеры),
есть ли уже токены/ключи.
Я дам точную навигацию по кабинету, что включить и где это находится, и проверочный чек‑лист доступов.

Где включить/найти Яндекс Доставку в кабинете
Зайдите в бизнес‑кабинет Яндекса под нужной организацией:
Если вас ведёт в Яндекс Go, это нормально — авторизация общая. В шапке проверьте, что выбрана ваша организация с ИНН 772863212942.
Откройте разделы, которые указывают на именно “Яндекс Доставку”:
Тарифы/договора: наличие договоров/оферты по “Доставке” (не только “Курьер Go”).
Интеграции/API: пункты вроде “Delivery API”, “Отгрузки”, “Склады/Точки забора”, “Временные окна/слоты”.
Документы/акты: доступность актов сдачи‑приемки, отчётности — это признак Яндекс Доставки.
Если видите только “Заявки/Claims”, “Курьеры”, “Маршруты Go” — это интерфейс Go Доставки. Ищите переключатель продукта/раздел “Доставка/Delivery” в меню. Иногда он появляется после подключения тарифа/заключения оферты.
Если нужного раздела нет:

Попросите администратора вашей организации выдать доступы к продукту “Яндекс Доставка” или подключить соответствующий тариф/оферту для этого ИНН.
Попробуйте войти в приватном окне и выбрать организацию с ИНН 772863212942 в селекторе.
Быстрый чек‑лист готовности к плановым доставкам
Договор/оферта по Яндекс Доставке: статус “действует”.
Точки отгрузки (склады) заведены с графиками работы и адресами.
Включены нужные тарифы и зоны обслуживания для вашего региона.
Настроены вебхуки на вашей стороне и сохранён их URL в кабинете.
Токен выпущен именно для Delivery API (а не только Go claims API).
Проверена калькуляция тарифов для типичных посылок.
Что проверить по токену
Тип токена: Bearer для Delivery API. Убедитесь, что им удаётся:
Рассчитать тариф: ответ содержит тарифы/ETA/стоимость.
Создать отгрузку/заказ: получает ID заказа/shipment.
Частая проблема: токен “видит” только Go API. Если на запросах к Delivery API получаете 401/403 или “unknown endpoint”, токен может быть выдан не на тот продукт. Решение — выпустить сервисный токен для Delivery или добавить нужные роли приложению.

Чтобы помочь вам дальше, вот как безопасно проверить, что токен подходит именно для Яндекс Доставки (плановые отгрузки), а не для Go:

Пошаговая проверка токена

Убедитесь, что это Bearer токен для Delivery API
Отправьте тестовый запрос калькуляции в Delivery API.
Если получите 401/403 — проверьте роли/продукт. Если 404/unknown endpoint — проверьте URL/версию API.
Пример запросов через curl
Замените на ваш токен. Выполняйте из безопасной среды.

Калькуляция:
bash
Copy

  -H 'Authorization: Bearer <TOKEN>' \
  -H 'Content-Type: application/json' \

    "pickup": {"address": "Москва, ул. Пример, 1", "datetime": "2025-11-01T14:00:00+03:00"},

    "items": [{"weight": 1.2, "length": 30, "width": 20, "height": 10, "price": 2500}]
  }'

Типичные ответы и что они значат

200/201: токен валиден, продукт и права подходят.
401 Unauthorized: токен неверен/протух — перевыпустить.
403 Forbidden: токен валиден, но нет прав на Delivery — выдать роли или выпустить токен для нужного продукта.
404/unknown endpoint: неверный URL/версия API либо вы обращаетесь к Go API с Delivery-путями (или наоборот).
Безопасность

Никогда не публикуйте токены в открытых чатах/репозиториях.
Храните в переменных окружения (DELIVERY_TOKEN) и прокидывайте в приложение.
Настройте ротацию токенов и оповещение при утечке.
Готов помочь проверить ответы ваших запросов (коды/тексты ошибок) и подсказать, как именно выдать недостающие права или какой раздел кабинета открыть, но, пожалуйста, перешлите только обрезанные логи без самого значения токена.
